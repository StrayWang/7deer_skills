# liuxiaopai-claude-workflow — 刘小排 AI 开发工作流

## 核心心法

> **「It is not for you, it is you.」**
> Claude Code 不是你的工具，它就是你自己的一部分。
> 不是问「它能帮我做什么」，而是「我要做什么，再把可以分包的部分交给它」。
> — 刘小排

**关键认知**：Claude Code 不只是编程工具，它是你的产品经理 + 工程师 + 测试 + 运维的综合体。只要是「手机和电脑能触碰到的虚拟世界」，只要能 SOP 化，就一定能被 Claude Code 自动化。

## 工作流全景

```
飞书/Notion 需求文档
        ↓
Claude Code 方案讨论（不直接写代码）
        ↓
ASCII 界面原型确认
        ↓
分模块实现（每次只做一个功能）
        ↓
自动化测试验证
        ↓
部署上线
        ↓
数据监控 → 迭代
```

## 第一步：写需求文档（最重要）

**刘小排铁律 #1：千万不要在 Claude Code 对话框里直接打字**

```markdown
# 需求文档模板（飞书/Notion 格式）

## 产品背景
- 目标用户：[具体描述，如「月收入 3000 美元以上的独立开发者」]
- 核心痛点：[用户现在怎么解决这个问题，有多痛]
- 我们的解法：[一句话描述]

## 功能需求
### 核心功能（MVP，必须有）
1. [功能名称]：[用户故事格式：作为 X，我希望 Y，以便 Z]
2. ...

### 次要功能（有更好，没有也能上线）
1. ...

## 技术约束
- 技术栈：[Next.js 14 / Python FastAPI / etc.]
- 数据库：[Supabase / PlanetScale / etc.]
- 部署平台：[Vercel / Railway / etc.]
- 预算限制：[月服务器成本上限]

## 验收标准
- [ ] 核心功能可以跑通
- [ ] 响应时间 < X 秒
- [ ] 移动端适配

## 参考竞品
- [竞品名] [URL]：[它好在哪，我们要超越哪里]
```

**为什么要写需求文档？**
- 避免歧义（Claude 会按字面意思理解，歧义 = 返工）
- 强迫自己想清楚再动手
- 文档即上下文，Claude Code 每次都能理解完整背景

## 第二步：方案讨论（不写代码）

```
第一轮对话模板：

「请阅读以下需求文档，先不要写任何代码。
告诉我：
1. 你对这个需求的理解是否正确？
2. 你建议的技术方案是什么？
3. 有哪些潜在风险或你不确定的地方？
4. 预计需要分几个阶段实现？

需求文档：[粘贴文档内容]」
```

**刘小排铁律 #2：先讨论方案，达成共识后再写代码**

这一步能避免 80% 的代码返工。

## 第四步：分模块实现

**刘小排铁律 #3：每次只做一个功能，做完测试通过再做下一个**

```python
# 错误做法（不要这样）
bad_prompt = """
帮我实现：
1. 用户登录
2. 图片上传
3. AI 处理
4. 结果展示
5. 支付功能
"""

# 正确做法（刘小排方式）
good_prompt_sequence = [
    "第一步：只实现用户登录功能，其他什么都不要做",
    "登录测试通过后，第二步：只实现图片上传功能",
    "上传测试通过后，第三步：只实现 AI 处理接口调用",
    # ...以此类推
]
```

## 第五步：自动化测试

```
刘小排的测试策略：让 Claude Code 自己写测试，自己跑测试

对话模板：
「功能实现后，请：
1. 写单元测试覆盖核心逻辑
2. 写一个端到端测试模拟用户完整操作路径
3. 自动运行测试并告诉我结果
4. 如果测试失败，自动修复并重新运行」
```

## 第六步：多模型协作（高阶用法）

刘小排的「三模型协作」工作流（2026年实践）：

```
Claude Code (Opus)  →  写开发计划 + 核心逻辑代码
Codex CLI (GPT)     →  审核代码 + 单元测试
Gemini CLI          →  设计界面 + 前端代码 + 端到端测试

分工逻辑：
- Claude：逻辑复杂度高、需要长上下文的任务
- GPT Codex：代码审查、测试生成（速度快）
- Gemini：UI/UX 设计、前端实现（多模态能力强）
```

## 产品成功率提升：刘小排「先有用户再做产品」法则

```
传统路径（失败率 95%）：
想法 → 做产品 → 找用户 → 失败

刘小排路径（成功率 90%+）：
洞察需求 → 验证需求（找 10 个人愿意付钱）→ 最小化实现 → 快速上线 → 迭代

验证需求的方法：
1. 在目标用户聚集的社区（Reddit / Twitter / Discord）发帖
2. 描述你要解决的问题，问「有没有人有这个痛点」
3. 收到 20+ 个「我也有这个问题」的回复 → 值得做
4. 收到 3 个人愿意预付费 → 立刻开始做
```

## 使用示例

```
用户输入：「我想做一个 AI 头像生成工具」

Agent 执行流程：
1. 触发需求文档模板，引导用户填写完整需求
2. 分析竞品（Raphael AI / PhotoAI 等）找差异化
3. 用「先有用户」法则验证需求可行性
4. 生成 MVP 功能清单（砍掉 80% 非必要功能）
5. 按刘小排工作流逐步实现：文档 → 方案 → ASCII → 分模块开发
6. 输出上线 checklist
```
